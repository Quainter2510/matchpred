# План: переход от «сайта ЧМ-2026» к платформе турниров прогнозов

> Рабочий документ. Фиксирует архитектуру пивота и пошаговый план. Первым
> реализуется **турнир по РПЛ**. Источник правды по текущему коду — AGENTS.md.

## Зафиксированные решения (из уточнений)

1. **Плоская модель.** «Комната» расширяется до «турнира» — отдельной сущности не заводим. Два одинаковых соревнования = две комнаты со своим типом/лигой/правилами.
2. **Туры — по календарной неделе.** Граница недели зависит от типа: РПЛ — среда, ЛЧ — суббота. Час границы переиспользуем (10:00 МСК). ЧМ остаётся на суточной границе.
3. **Спецпрогноз РПЛ «лидер лиги» — вручную админом** (как сейчас итоговый бомбардир на ЧМ). Автонагрузки из /standings не делаем.
4. **Лига — от типа, сезон — от админа.** РПЛ→league_id 235, ЛЧ→2, ЧМ→1 (зашиты в коде). Сезон админ выбирает при создании.
5. **Длительность — туры + даты.** UI показывает список туров и даты; выбор «с тура N по тур M» проставляет окно дат, по которому фактически отбираются матчи.
6. **ЧМ сохраняется как тип `world_cup`.** Существующие комнаты миграцией получают этот тип и продолжают работать без изменений поведения.

## Ключевой архитектурный сдвиг

Сейчас нарушается главный инвариант «матчи глобальны и общие для всех комнат»:
теперь у каждого турнира **своя реальная лига+сезон** и **свой пул матчей**.

- `matches` остаётся общей дедуплицированной таблицей (по `api_football_id`), но каждый матч получает метки `league_id`, `season`, `round`.
- «Матчи турнира» = резолвер:
  - лиговые типы (ЧМ/РПЛ/ЛЧ): `league_id = room.league_id AND season = room.season` (+ окно `[starts_on, ends_on]` у РПЛ/ЛЧ);
  - кастомный тип (топ-5+РПЛ+ЛЧ): явный список в join-таблице `tournament_matches` (вводится позже, фаза 3).
- Результат матча по-прежнему **глобален** (инвариант #9 сохраняется): один ввод счёта затрагивает все турниры над этой лигой.

Единая точка `services/tournament.py::tournament_matches_filter(room)` — SQLAlchemy-фильтр, через который проходят ВСЕ чтения матчей/прогнозов/таблиц/положения. Это самый крупный рефактор: каждый запрос по матчам скоупится турниром, а не «всеми матчами».

## Модель данных

### `rooms` — расширение (миграция 0012)

| Поле | Тип | Назначение |
|------|-----|-----------|
| `tournament_type` | VARCHAR(20) NOT NULL, default `world_cup` | `world_cup│rpl│ucl│custom` |
| `league_id` | INT NULLABLE | API-Football лига (1/235/2…); NULL у custom |
| `season` | INT NULLABLE | сезон API |
| `tour_anchor` | SMALLINT NULLABLE | день недели-начало тура (0=Пн…6=Вс); NULL = суточная группировка (ЧМ) |
| `starts_on` / `ends_on` | DATE NULLABLE | окно включения матчей (длительность); NULL/NULL = весь турнир (ЧМ) |
| `special_kind` | VARCHAR(20) NOT NULL, default `wc` | `wc`(чемпион+бомбардир) `│leader│stage_or_champion│none` |
| `special_result_team` | VARCHAR(100) NULLABLE | вручную заданный ответ спецпрогноза (лидер РПЛ и т.п.) |

Существующие `points_champion` переиспользуем как «очки за команду-спецпрогноз» (чемпион/лидер); `points_scorer` — только для ЧМ. `first_match_at` остаётся дедлайном спецпрогноза.

**Бэкфилл ЧМ:** всем текущим комнатам — `tournament_type='world_cup'`, `league_id=1`, `season=2026`, `tour_anchor=NULL`, `special_kind='wc'`.

### `matches` — метки лиги (миграция 0012)

| Поле | Тип | Назначение |
|------|-----|-----------|
| `league_id` | INT NULLABLE | лига API (бэкфилл существующих = 1) |
| `season` | INT NULLABLE | сезон (бэкфилл = 2026) |
| `round` | VARCHAR(40) NULLABLE | сырой `round` из API (для выбора «с тура по тур») |

Индекс `(league_id, season, kickoff_at)`. `api_football_id` остаётся UNIQUE (дедуп).
`match_date` теперь = **метка тура по схеме лиги** (неделя для РПЛ/ЛЧ, день для ЧМ), считается при синхронизации.

### `special_predictions` — обобщение

`champion_team` становится универсальным «выбор команды» (чемпион ЧМ / лидер РПЛ / стадия ЛЧ). `top_scorer_*` используются только типом `wc`. Схема таблицы не меняется — меняется семантика и UI.

### `tournament_matches` (фаза 3, для custom)

`(room_id, match_id)` — явный набор матчей, отобранных админом. Для лиговых типов не используется (набор выводится резолвером).

## Риски и на что смотреть

- **Скоуп чтений (фаза 0)** — самое опасное: пропустить хоть один запрос по матчам без скоупа = утечка чужой лиги в турнир. Резолвер + прогон всех роутеров обязателен.
- **Идемпотентность пересчёта** (инвариант #9) — сохранить: правки `recalc.py` не должны ломать снятие/начисление.
- **Мульти-лиговый синк и квоты API-Football** — тик теперь бьёт по нескольким лигам; следить за лимитом запросов, `_should_poll` должен резать пустые лиги.
- **Симуляция** (`simulation.py`) — фейковые счета/туры должны уважать схему тура турнира.
- **Бэкфилл ЧМ** — проверить, что существующие комнаты и их матчи корректно получили `league_id=1/season=2026` и `match_date` не переехал (у ЧМ схема суточная — значения совпадают).
